# 开源发布准备清单

这份清单面向 FilePilot 当前仓库状态，目标是在公开仓库前，把“能不能发”“别人能不能跑”“出了问题怎么协作”这三件事补齐。

建议按下面顺序推进，不要一上来只改 `README.md`。

## 当前已决定

- 许可证采用 `MIT`
- 当前先公开内部协作文件，后续再按需要收敛
- 配置入口的公开叙事仍待确认

## 一、先做发布决策

### 1. 明确公开范围

- 决定哪些内容属于公开仓库，哪些内容继续留在私有环境。
- 当前仓库里建议重点复查这些目录和文件：
  - `.claude/`
  - `.spec-workflow/`
  - `AGENTS.md`
  - `CLAUDE.md`
  - `docs/superpowers/`
  - `docs/` 下偏评审、修复总结、内部设计流转的文档

建议：

- 用户无感、仅服务内部协作的文件，不要默认公开。
- 如果要保留，至少要在文档中说明它们是研发辅助材料，而不是用户入口。

### 2. 选择许可证

- 在仓库根目录新增 `LICENSE`。
- 在发布前决定是否允许商业使用、修改分发、闭源集成。

常见选择：

- `MIT`：最宽松，适合工具类项目快速传播。
- `Apache-2.0`：比 MIT 多了专利授权条款，更稳妥。
- `GPL-3.0`：更强 copyleft，适合你明确希望衍生版本继续开源时使用。

## 二、补齐开源仓库基础文件

这些文件是公开仓库的基础设施，优先级很高。

### 必需

- `LICENSE`
- `README.md`
- `CONTRIBUTING.md`
- `SECURITY.md`

### 强烈建议

- `CODE_OF_CONDUCT.md`
- `CHANGELOG.md`
- `.github/ISSUE_TEMPLATE/`
- `.github/pull_request_template.md`

## 三、整理 README 的公开定位

公开版 `README` 不应该写成内部研发手册，也不应该只写技术栈。

建议保留：

- 项目一句话介绍
- 适合谁使用
- 核心功能
- 当前支持平台
- 最短安装和启动步骤
- 截图或演示 GIF
- 当前限制
- 相关文档入口

建议删减或下沉：

- 运行时内部字段
- 详细目录结构解释
- 内部设计约束细节
- 面向开发者的全部排障过程
- 过多阶段性方案和历史评审说明

建议单独补一节“当前限制”：

- 当前以 Windows 环境为主
- 桌面壳与文件夹图标能力偏 Windows
- 需要用户自行配置模型服务
- 不是云端托管服务

## 四、统一配置与安装故事

当前仓库同时存在：

- `config.example.json`
- `.env.example`

但代码主路径明显更偏向 `config.json`。

发布前建议统一成一种主叙事：

- 推荐方案：以 `config.json` 为主配置入口
- `.env.example` 只保留为兼容方式或高级部署方式

需要补充的文档说明：

- 首次启动最少需要填哪些配置
- 哪些字段是必填
- 哪些功能没配置也能运行
- API Key 存在哪里
- 配置文件不会提交到版本控制

## 五、补齐隐私与安全说明

这是这个项目开源时非常重要的一块，因为它会处理本地文件并调用外部模型接口。

至少要在 `README` 或 `SECURITY.md` 中明确：

- 项目默认在本地处理文件
- 哪些信息会发送到外部模型服务
- 用户需要自行提供 API Key
- 作者不提供托管模型服务
- 使用前应避免把敏感目录直接交给外部模型分析

如果后续要做公开主页或发布页，建议再补一份独立的隐私说明。

## 六、清理仓库内容

发布前做一次“公开视角清扫”。

### 1. 清理不应公开的内容

- 本地调试产物
- 临时输出目录
- 私有流程文档
- 与个人工作流强绑定的辅助文件

### 2. 检查历史提交

重点确认这些内容从未进入 git 历史：

- `config.json`
- 真实 API Key
- 本地路径
- 调试日志
- 运行时输出
- 测试时生成的样本数据

### 3. 检查 `.gitignore`

确认这些路径被覆盖并且实际未被误提交：

- `output/`
- `logs/`
- `frontend/out/`
- `frontend/.next/`
- `desktop/src-tauri/target/`
- `.env`
- `config.json`

## 七、补自动化验证

现在仓库里有桌面打包 workflow，但还不够覆盖日常公开协作。

建议至少增加一个基础 CI，在 `push` / `pull_request` 时自动运行：

- Python `unittest`
- 前端 `npm run typecheck`
- 前端 `npm test`
- `desktop/src-tauri` 下的 `cargo check`

这样开源后别人提 PR，不会只能靠人工判断是否可合并。

## 八、准备首次公开版本

建议在第一次公开前完成一次“从零环境安装演练”。

核对项：

1. 一个陌生用户只看 `README`，能否完成安装。
2. 没有本地历史配置时，是否能理解如何配置模型。
3. 至少能跑通一个最小功能链路。
4. 报错信息是否足够清楚。
5. 缺失配置时，界面和文档是否一致。

建议首次公开时同步准备：

- 一个 `v0.x` release
- 一份简短发布说明
- 一张主界面截图
- 一个最小演示流程

## 九、按仓库现状的建议优先级

### P0：公开前必须完成

- 选择并添加 `LICENSE`
- 清理或移动内部协作文件
- 统一配置说明
- 重写公开版 `README`
- 补 `SECURITY.md`

### P1：建议首个公开版本前完成

- 补 `CONTRIBUTING.md`
- 补 `CODE_OF_CONDUCT.md`
- 增加基础 CI
- 增加截图或 GIF
- 补 issue / PR 模板

### P2：公开后尽快补齐

- `CHANGELOG.md`
- 更细的使用文档
- FAQ
- 示例配置与示例流程

## 十、建议的实际执行顺序

1. 决定许可证和公开范围。
2. 清理内部文件、私有文档和潜在敏感内容。
3. 统一配置入口和首次启动说明。
4. 重写公开版 `README`。
5. 补 `LICENSE`、`SECURITY.md`、`CONTRIBUTING.md`。
6. 增加 CI 和 issue / PR 模板。
7. 从零环境做一次完整安装演练。
8. 打第一个公开 release。
