# 2026-03-19 文件读取端增强设计

## 概述

本设计聚焦文件读取端的三个增强方向：

1. 图像解析能力
2. 压缩包预览能力
3. 文本编码鲁棒性

目标是在不破坏现有分析 -> 扫描落盘 -> 整理命令生成主链路的前提下，提升对中文 Windows 本地文件的判断质量。

本次设计优先强调兼容性和风险隔离，不直接进入实现。

## 当前现状

当前读取端主要由以下两类能力组成：

- `list_local_files(...)`：输出目录浅层摘要
- `read_local_file(...)`：按文件类型读取文本或摘要，并以字符串形式返回给分析阶段工具调用

分析阶段依赖这些工具返回的字符串继续驱动主模型完成 `<output>` 生成，整理阶段再消费扫描文本完成命令流校验。因此，读取端的返回契约不能被随意改写。

## 已确认的设计决策

### 1. 图像解析只采用 Vision 路线

不引入本地 OCR，不接入 `pytesseract`，原因如下：

- 额外安装成本较高，Windows 本地环境体积不划算
- OCR 质量对截图、界面图、混排内容不稳定
- 对本项目场景，Vision 模型的语义理解价值更高

### 2. 图片分析使用全局独立配置

图片分析不复用主分析模型配置，而是使用单独的一组配置：

- `IMAGE_ANALYSIS_ENABLED`
- `IMAGE_ANALYSIS_BASE_URL`
- `IMAGE_ANALYSIS_API_KEY`
- `IMAGE_ANALYSIS_MODEL`

设计目标是允许主分析端点不支持图像输入，而图片分析单独走另一个支持 Vision 的服务。

### 3. 图片分析采用严格模式

不做自动降级，不回退 OCR，不回退到主分析 Agent 猜测。

如果图片分析配置缺失、模型不支持图像、调用失败或响应异常，直接返回明确失败信息，交由主分析链路继续处理。

### 4. 图片分析必须与主分析上下文隔离

当读取端遇到图片文件时，不把图片直接交给当前分析阶段 Agent。

而是：

- 发起一个新的、无历史消息的独立图片分析调用
- 仅传入固定 system 指令、图片本身和一个简单用户问题
- 返回一段简短自然语言摘要

这样可以避免主分析上下文污染，降低链路耦合度。

### 5. 图片分析返回简短自然语言摘要

返回结果不要求结构化标签，也不要求长说明。

目标样式类似：

- “这是一张聊天截图，主要在讨论付款和合同修改。”
- “这是一张软件报错截图，界面显示数据库连接失败。”
- “这是一张票据照片，看起来像报销发票。”

## 推荐架构

### 图像能力

新增独立模块：

- `file_organizer/analysis/image_describer.py`

建议职责：

- 读取图片分析相关配置
- 校验 Vision 配置完整性
- 发起独立图片分析请求
- 把模型返回压缩成简短自然语言摘要
- 在失败时返回统一错误消息

`file_reader.py` 仅负责：

- 判断文件后缀是否属于图片
- 调用 `describe_image(...)`
- 将结果继续包装为当前工具调用兼容的字符串格式

### 压缩包预览能力

建议新增工具能力：

- `read_archive_index(...)`

第一阶段仅支持：

- `.zip`

第一阶段不建议直接支持：

- `.rar`
- `.7z`

原因：

- `zipfile` 为 Python 标准库，稳定且零新增系统依赖
- `.rar` 与 `.7z` 常需要额外依赖或系统程序，复杂度明显上升

压缩包工具建议只输出：

- 顶层目录
- 部分关键文件名
- 文件数摘要
- 目录树的前若干条

不做解压落盘，不读取内部文件内容，不改变当前执行链路。

### 编码鲁棒性

建议优先采用固定顺序 fallback，而不是一开始引入 `chardet`：

1. `utf-8`
2. `utf-8-sig`
3. `gbk`
4. `utf-16`

只有这些都失败时，才返回“可能为二进制或未知编码”。

原因：

- 更可控
- 对中文 Windows 常见文件覆盖度已经较高
- 避免 `chardet` 在短文本或混合编码场景下误判

后续如真实数据中编码误判频繁，再考虑引入 `chardet` 作为增强项。

## 对现有功能的影响分析

### 低风险变更

- 为普通文本增加编码 fallback
- 为读取端新增结构化内部 helper，但继续保留字符串输出契约
- 为 `.zip` 增加只读索引工具
- 为图片文件新增独立 Vision 摘要读取分支

这些变更都可以在不改变当前分析阶段主对话结构的前提下完成。

### 中风险变更

- 将路径权限重新收紧到目标目录作用域
- 为多厂商 Vision 端点做统一兼容

这些增强值得做，但需要额外测试不同配置和错误分支。

### 高风险变更

- 把 `read_local_file(...)` 的返回从字符串改成结构化对象
- 改变 `list_local_files(...)` 的默认扫描深度或默认输出语义
- 把图片直接塞进主分析 Agent 当前上下文

这些变更可能直接影响分析输出、扫描校验和整理命令流验证，不应在同一轮增强中进行。

## 配置设计建议

建议在共享配置层新增图片分析配置读取逻辑，并提供单独客户端工厂。

推荐环境变量：

- `IMAGE_ANALYSIS_ENABLED=true`
- `IMAGE_ANALYSIS_BASE_URL=...`
- `IMAGE_ANALYSIS_API_KEY=...`
- `IMAGE_ANALYSIS_MODEL=...`

设计要求：

- 不影响主分析模型配置
- 配置缺失时可明确报错
- 不做隐式回退

## 调用行为建议

### 图片分析请求

建议发送：

- 一条固定 system 指令：要求只返回简短自然语言摘要，用于文件整理判断
- 一条 user 请求：请简要描述图片主要内容
- 一张图片输入

设计原则：

- 不携带主分析阶段任何历史消息
- 不带目录上下文、已有扫描结果或推理上下文
- 不要求模型给出分类标签或过长解释

### 工具返回格式

为了兼容现有工具消息契约，读取端对主分析 Agent 的返回仍为字符串，例如：

`--- 文件 [xxx.png] 内容开始 ---`
`这是一张聊天截图，主要在讨论付款和交付时间。`
`--- 内容结束 ---`

这样可以最大化减少对现有分析逻辑的冲击。

### 工具说明同步更新

新增或增强读取能力时，必须同步更新以下说明层，避免模型仍按旧理解调用工具：

- `analysis/service.py` 中的 `tools` 描述
- `analysis/prompts.py` 中对可用工具与适用场景的说明
- `README.md` 中对读取端能力的简要介绍

对于图片分析与压缩包预览，工具说明应明确：

- 图片分析走独立 Vision 调用，不继承主分析上下文
- 压缩包只读取索引，不解压、不读取内部正文
- 文本读取会尝试常见中文 Windows 编码 fallback

## 实施建议顺序

推荐顺序：

1. 编码鲁棒性增强
2. `.zip` 压缩包索引工具
3. 图片 Vision 独立分析能力
4. 进一步抽取读取端结构化 helper

原因：

- 前两项最稳，收益明确
- 图片能力价值最大，但外部接口和错误处理更复杂
- 先把基础读取稳定性做好，再扩展语义读取能力，风险更可控

## 测试建议

### 编码增强

- UTF-8 文本
- UTF-8 with BOM
- GBK 文本
- UTF-16 文本
- 二进制文件误读失败

### 压缩包索引

- 普通 zip 文件
- 含多层目录的 zip 文件
- 大 zip 的预览截断
- 非 zip 文件输入报错

### 图片分析

- 配置缺失时报错
- Vision 调用成功返回简短摘要
- Vision 调用失败时严格报错
- 主分析链路仍收到字符串形式结果
- 图片分析调用不继承主分析历史消息

## 暂不纳入本轮的内容

- 本地 OCR
- `.rar` / `.7z` 深度支持
- 改写分析阶段主会话协议
- 改变扫描输出格式
- 增加图片分类标签体系

## 总结

本设计的核心原则是：

- 新增能力，不替换现有契约
- 图片分析独立配置、独立上下文、严格失败
- 压缩包先做 zip 索引
- 编码优先做固定 fallback
- 工具说明与提示词必须同步更新

这样可以在保持当前主流程稳定的前提下，逐步把读取端从“文本读取器”升级为“文件内容探查器”。
