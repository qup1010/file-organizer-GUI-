# 设置页预设创建与编辑流重构设计

## 背景

当前设置页中，`文本模型`、`图片理解`、`图标生图` 三个分组都采用“预设切换 + 下方直接编辑表单”的结构，但存在两个明显问题：

1. 表单内的“预设名称”字段与顶部预设切换器重复，信息层级不清。
2. 首次使用时，用户可能直接填写接口地址、模型和 API Key，然后点击“保存”，但由于当前配置并不是一个可持久化的用户预设，保存无法按用户预期完成，必须先额外执行“新建预设”。

这会让用户在最关键的首次配置阶段遇到阻断，且阻断出现得太晚。

## 目标

1. 统一 `文本模型`、`图片理解`、`图标生图` 三个分组的交互模型。
2. 将“先创建预设，再编辑配置”前置为显式流程，避免先填后失败。
3. 移除表单中重复的“预设名称”字段。
4. 让 API Key、模型、接口地址等字段始终落到一个明确存在的用户预设上。
5. 保持现有预设切换、删除、测试连接、保存配置等核心能力不变。

## 非目标

1. 本次不改动 `抠图` 分组的交互模式。
2. 本次不引入“表单内直接重命名预设”能力。
3. 本次不调整后端预设数据模型，只重构前端交互与提交流程。

## 方案选型

### 方案 A：点击 `+` 后先弹出命名对话框，再进入编辑

这是推荐方案。

流程如下：

1. 用户在分组顶部看到预设切换器和 `+` 按钮。
2. 点击 `+` 后弹出一个小对话框，只要求输入预设名称。
3. 创建成功后，自动激活该预设。
4. 下方表单切换到这个新预设的编辑态，用户继续填写模型、接口地址、API Key 等内容。
5. 后续点击“保存”时，保存的是当前激活预设的内容。

优点：

1. 用户先建立“我要创建一个配置”的明确意图，再开始填写细节，心智模型最清晰。
2. 预设名称只出现一次，不会和表单内容重复。
3. 避免用户在大量输入后，才发现当前状态不可保存。

缺点：

1. 首次创建比直接展开空白表单多一步对话框确认。

### 方案 B：点击 `+` 后直接展开空表单，保存时再补预设名

不推荐。虽然减少一步弹框，但会重新引入“先填内容，后补名称”的迟滞反馈问题。

### 方案 C：系统自动生成预设名

不推荐。虽然创建速度最快，但会快速积累低质量命名，降低长期可维护性。

## 最终设计

### 1. 顶部预设区职责收敛

`文本模型`、`图片理解`、`图标生图` 三个分组的顶部区域统一承担以下职责：

1. 查看当前激活预设。
2. 切换预设。
3. 点击 `+` 新建预设。
4. 删除可删除的用户预设。

顶部区域不再和下方表单一起承担“创建时临时命名”的职责。

### 2. 新建入口统一为 `+`

三个分组统一使用 `+` 作为主新建入口，视觉上与预设切换器紧邻，表示“先有预设，再有配置编辑”。

当用户点击 `+` 时：

1. 打开一个轻量对话框。
2. 对话框只展示预设名称输入框与确认/取消操作。
3. 确认后调用现有创建预设接口。
4. 创建成功后自动刷新设置快照并激活新预设。

### 3. 配置表单移除“预设名称”

以下分组表单移除“预设名称”字段：

1. `文本模型`
2. `图片理解`
3. `图标生图`

表单仅保留真正属于配置内容的字段，例如：

1. 模型 ID
2. 接口地址
3. API Key
4. 生图尺寸、并发、保存方式等

预设名称的展示以顶部切换器为准。

### 4. 首次使用的页面状态

当某个分组还没有可编辑的用户预设时，页面不应允许用户直接在一个不可持久化的空状态上编辑并点击保存。

界面行为应为：

1. 顶部仍然显示预设区和 `+` 按钮。
2. 下方表单区域显示提示态，说明“请先点击 `+` 创建一个预设”。
3. 用户创建成功前，不展示或不启用详细编辑表单。

这样用户从一开始就知道保存对象是什么，不会误以为“当前这块空表单就是一个可直接保存的配置”。

### 5. 已有预设时的编辑行为

当当前分组已有激活预设时：

1. 表单正常展示该预设的可编辑字段。
2. 用户修改后点击“保存”，沿用现有保存接口更新当前激活预设的配置。
3. API Key 的保存逻辑绑定当前激活预设，不再依赖额外先建一次预设。

### 6. 三个分组行为完全一致

下列行为必须在三个分组中完全统一：

1. 新建入口位置
2. 新建流程
3. 无可编辑预设时的提示态
4. 创建成功后的自动激活
5. 表单是否展示
6. 保存目标的语义

避免出现一个分组能直接编辑，另一个分组必须先新建的割裂体验。

## 数据流与状态调整

前端页面状态需要做以下调整：

1. 当前激活预设是否为可编辑用户预设，需要有明确判断。
2. “创建预设”对话框与“新建后进入编辑态”的状态要区分清楚。
3. 构建保存 payload 时，不再依赖表单中的 `name` 字段。
4. 连接测试同样应基于当前激活预设的字段，而不是一个未落盘的临时命名状态。

如果现有 `default` 预设仍保留作为基础模板，则前端需要区分：

1. 是否允许编辑 `default`
2. 是否允许在 `default` 上直接保存密钥

本次设计建议：首次配置场景不要把“直接写入 default”作为主路径，而是引导用户创建自己的预设。

## 错误处理

### 创建预设失败

1. 关闭 loading。
2. 保持对话框打开。
3. 展示错误信息。
4. 不进入表单编辑态。

### 保存失败

1. 保持当前表单输入不丢失。
2. 展示保存失败信息。
3. 不回滚用户已输入内容。

### 删除当前预设

沿用现有删除逻辑，但需要保证删除后页面不会停留在一个无效的编辑态。若回退到不可编辑状态，应展示统一提示态。

## 测试策略

### 前端交互

至少覆盖以下行为：

1. 点击 `+` 后弹出命名对话框。
2. 创建成功后自动激活新预设。
3. 三个分组不再渲染“预设名称”字段。
4. 无可编辑预设时，表单隐藏或禁用，并显示引导提示。
5. 已有激活预设时，编辑并保存可正常调用保存接口。

### 回归验证

至少验证以下现有能力未被破坏：

1. 预设切换
2. 删除预设
3. 连接测试
4. API Key 保存
5. 图标生图专有字段保存

## 实施边界

本次实现优先聚焦在：

1. `frontend/src/app/settings/page.tsx`
2. 相关设置页子组件或类型定义
3. 与该页面强相关的前端测试

如后端接口已经满足“先创建预设，再更新预设”的路径，则不需要扩大到服务端改造。

## 验收标准

满足以下条件即可视为完成：

1. `文本模型`、`图片理解`、`图标生图` 三个分组均不再显示表单内“预设名称”字段。
2. 三个分组都通过顶部 `+` 按钮创建预设。
3. 点击 `+` 后先弹出预设名输入对话框。
4. 预设创建成功后自动激活并进入编辑态。
5. 首次使用时，用户不会再出现“已经填写 API Key，但因未先新建预设而无法保存”的情况。
6. 现有切换、删除、测试、保存流程保持可用。
